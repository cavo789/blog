**Project Name:** DocuVerse

**Project Goal:**
Create a high-performance, containerized, on-the-fly Markdown documentation server. The application will serve a beautiful, responsive, and feature-rich web interface to browse and read Markdown files from a read-only directory. The architecture must be decoupled into a Python/FastAPI backend, a React/Next.js frontend, a Redis cache, and a Meilisearch engine for powerful search capabilities, all orchestrated via Docker Compose.

### **1. High-Level Architecture & Services**

The application will consist of four distinct services running in Docker containers:

1.  **`backend` (Python/FastAPI):** The core API. It handles file system scanning, Markdown-to-HTML conversion, caching logic with Redis, and indexing/searching with Meilisearch.
2.  **`frontend` (Next.js/React):** The user-facing web application. It consumes the backend API to display the file structure, rendered content, and search results. It must be styled with **Tailwind CSS**.
3.  **`redis` (Official Image):** A high-speed in-memory cache to store the generated HTML, avoiding redundant Markdown parsing.
4.  **`meilisearch` (Official Image):** A powerful search engine to provide instant, fuzzy, and typo-tolerant search across all documentation files.

### **2. Detailed Backend Specification (`backend` service)**

*   **Framework:** **Python 3.11** with **FastAPI**.
*   **Markdown Parsing:** Use the `markdown-it-py` library. It must be configured with plugins for:
    *   Syntax highlighting for code blocks (e.g., using `pygments`).
    *   Automatic anchor links for headings (`<h2>`, `<h3>`, etc.).
*   **Caching Strategy:**
    1.  When a request for a file arrives, the backend MUST read the raw content of the `.md` file.
    2.  It MUST compute a **SHA256 hash** of the file's content.
    3.  It MUST query Redis using the hash as the key (e.g., `GET doc-sha256:<hash>`).
    4.  **Cache Hit:** If Redis returns HTML, serve it immediately.
    5.  **Cache Miss:** If Redis returns `nil`, parse the Markdown to HTML, store the result in Redis (`SET doc-sha256:<hash> <html_content>`), and then serve it.
*   **Search Integration (Meilisearch):**
    *   **On Startup:** The backend MUST recursively scan the entire `/docs` directory. For each `.md` file found, it must read its content, strip the Markdown to plain text, and send it to a Meilisearch index named `docs`. Each document in Meilisearch should have at least two fields: `filePath` (e.g., `getting-started/installation.md`) and `content`.
*   **API Endpoints:**
    *   `GET /api/file-tree`: Returns a JSON object representing the recursive directory structure of the `/docs` volume. Exclude non-Markdown files.
        *   *Example Response:* `{ "name": "docs", "type": "folder", "children": [ { "name": "guide.md", "type": "file", "path": "guide.md" }, { "name": "api", "type": "folder", "children": [...] } ] }`
    *   `GET /api/render/{file_path:path}`: Renders a specific Markdown file.
        *   It implements the caching logic described above.
        *   The response must be a JSON object containing the rendered HTML and a Table of Contents structure.
        *   *Example Response:* `{ "htmlContent": "<h1>Title</h1>...", "toc": [{ "level": 1, "text": "Title", "id": "title" }, { "level": 2, "text": "Subtitle", "id": "subtitle" }] }`
    *   `POST /api/search`: Performs a search query against Meilisearch.
        *   *Request Body:* `{ "query": "fuzzy search boolean" }`
        *   *Response:* A JSON array of results, including the file path and a content snippet (highlighting the match). `[ { "filePath": "...", "snippet": "..." } ]`
*   **Static Files:** The backend must be able to serve images or other assets referenced in the Markdown files from the `/docs` directory. Create an endpoint like `GET /api/assets/{file_path:path}`.

### **3. Detailed Frontend Specification (`frontend` service)**

*   **Framework:** **Next.js** with **React** and **TypeScript**.
*   **Styling:** **Tailwind CSS**. The UI must be clean, modern, and prioritize content readability.
*   **Layout (Three-column design):**
    1.  **Left Sidebar (File Explorer):** Displays the file tree fetched from `/api/file-tree`. It should be collapsible.
    2.  **Center Column (Content):** Displays the `htmlContent` returned by `/api/render`. This area should take up the maximum available space.
    3.  **Right Sidebar (Table of Contents):** Displays the active page's ToC, generated from the `toc` object from the API. It should highlight the current section as the user scrolls.
*   **Persistent UI Elements:**
    *   **Header:** A slim header at the top, always visible. It contains the **Search Bar** and a **Dark/Light Mode toggle**.
    *   **Breadcrumbs:** Located just above the content, showing the path to the current file (e.g., `Home > API Reference > Endpoints`).
*   **Core Features:**
    *   **Dark/Light Mode:** Must respect the user's OS preference (`prefers-color-scheme`) by default but be overridable with the toggle. Use Tailwind's `dark:` variant.
    *   **Search UI:** The search bar in the header should trigger a call to `/api/search`. Results should be displayed in a clean overlay or dropdown. Clicking a result navigates to that page. The search must support fuzzy search and boolean operators as provided by Meilisearch.
    *   **Code Blocks:** All code blocks (`<pre><code>`) must have:
        *   Excellent syntax highlighting (use `highlight.js` or a similar library).
        *   A "Copy to Clipboard" button that appears on the top-right corner on hover.
    *   **Printing:** The application must have a print-specific stylesheet (`@media print`). When a user hits CTRL+P, the printed output MUST be a clean, perfectly formatted document containing only the main content (no sidebars, headers, etc.).
*   **Customization:**
    *   The application must load an optional `/config/custom.css` file if it exists. This allows users to override the default styles.

### **4. Containerization & Deployment**

*   **`Dockerfile` for `backend`:** Must be a **multi-stage build** to keep the final image size small. Use Poetry for dependency management.
*   **`Dockerfile` for `frontend`:** Must also be a **multi-stage build**, compiling the Next.js app into an optimized production server.
*   **`compose.yaml`:**
    *   Define all four services (`backend`, `frontend`, `redis`, `meilisearch`).
    *   Configure service dependencies (`backend` depends on `redis` and `meilisearch`).
    *   **Volumes:**
        *   Mount the user's documentation folder as a read-only volume: `- ./my-docs:/docs:ro`.
        *   Mount an optional config folder: `- ./config:/config:rw`.
        *   Use named volumes for Redis (`redis_data`) and Meilisearch (`meili_data`) data persistence.
    *   Expose frontend on port `3000` and Meilisearch on `7700`.
