# C5 — AI-powered "Explain Like I'm Five" tooltips in Snippet

**Status:** Plan approved, not yet implemented
**Estimated effort:** ~5–7h (script + component renderer + CSS + wiring)

---

## What this feature does

A `?` badge appears at the right edge of selected code lines inside `<Snippet>` blocks. Hovering (or tapping on mobile) the badge reveals a short, plain-language tooltip that explains what that line does — targeted at junior readers who may not be familiar with Dockerfiles, YAML anchors, shell idioms, etc.

```
  ┌─ Dockerfile ──────────────────────────────────────── ▾ ─┐
  │  1  FROM node:20-alpine                             [?]  │
  │  2                                                       │
  │  3  WORKDIR /app                                         │
  │  4  COPY package*.json ./                          [?]  │
  │  5  RUN npm ci --omit=dev                          [?]  │
  │  6                                                       │
  │  7  COPY . .                                             │
  │  8  EXPOSE 3000                                    [?]  │
  │  9  CMD ["node", "server.js"]                      [?]  │
  └──────────────────────────────────────────────────────────┘
           ↓ hover [?] on line 5
  ┌──────────────────────────────────────────────────────────┐
  │  npm ci installs exactly what's in package-lock.json.   │
  │  --omit=dev skips dev tools — keeps your image smaller. │
  └──────────────────────────────────────────────────────────┘
```

**Design principles:**
- Explanations are generated **at build time** by a Node script calling the Claude API. No API key ever reaches the browser, and hover latency is zero.
- Claude decides which lines are non-trivial — blank lines, closing braces, and obvious assignments are skipped.
- The feature is **opt-in per snippet**: the author imports the `.eli5.json` file and passes it as a prop.
- Works on Dockerfile, `docker-compose.yaml`, `devcontainer.json`, `.env`, shell scripts, YAML, TOML — any file type the Snippet component already handles.

## Architecture overview

```
  blog/2026-01-01-my-post/
  ├── index.mdx               ← imports eli5 JSON
  ├── Dockerfile              ← source file shown in Snippet
  └── Dockerfile.eli5.json   ← generated by the script
```

```
  yarn eli5 blog/2026-01-01-my-post/Dockerfile
       ↓
  scripts/generate-eli5.mjs  →  Claude API (claude-haiku-4-5-20251001)
       ↓
  Dockerfile.eli5.json written
       ↓
  Author adds import + eli5 prop to MDX
       ↓
  yarn build / yarn start
       ↓
  Snippet renders line-by-line with ? badges
```

## Files to create

### `scripts/generate-eli5.mjs`

CLI entry point. Reads a source file, calls Claude, writes `<file>.eli5.json`.

**Usage:**
```bash
node scripts/generate-eli5.mjs <source-file> [--force] [--output <path>]

# Examples
node scripts/generate-eli5.mjs blog/2026-01-01-my-post/Dockerfile
node scripts/generate-eli5.mjs blog/2026-01-01-my-post/compose.yaml --force
node scripts/generate-eli5.mjs blog/2026-01-01-my-post/.env --output blog/2026-01-01-my-post/.env.eli5.json
```

**Behavior:**
1. Check if `<source-file>.eli5.json` already exists → skip unless `--force`
2. Read `ANTHROPIC_API_KEY` from `process.env` (or `.env` at repo root via `dotenv`)
3. Number each line of the source file (1-indexed)
4. Call Claude with a carefully structured prompt (see below)
5. Validate the response is valid JSON with string keys and string values
6. Write the JSON file with metadata header

**Output format (`Dockerfile.eli5.json`):**
```json
{
  "version": 1,
  "model": "claude-haiku-4-5-20251001",
  "generated": "2026-06-13T10:00:00.000Z",
  "source": "Dockerfile",
  "lang": "dockerfile",
  "explanations": {
    "1": "FROM picks the base image — the pre-built foundation your image starts from. node:20-alpine is Node 20 on Alpine Linux, a very small OS.",
    "4": "COPY package*.json ./ copies only the dependency manifest files first, so Docker can cache the npm install step separately.",
    "5": "npm ci is like npm install but strictly follows package-lock.json. --omit=dev leaves out development-only packages, shrinking the final image.",
    "8": "EXPOSE documents which port the container listens on. It's a label — it doesn't actually open the port; docker run -p does that.",
    "9": "CMD is the default command that runs when the container starts. Square-bracket form (exec form) runs node directly without a shell wrapper."
  }
}
```

**Claude prompt (system):**
```
You are a patient senior developer explaining code to a junior colleague.
Given a source file with numbered lines, identify only the lines that a junior
developer might find confusing. For each such line, write exactly one or two
plain-English sentences — friendly, concrete, no jargon unless you immediately
define it. Think "explain like I'm five but I'm also a developer".

Skip: blank lines, closing braces/brackets, import lines that are self-evident,
comments that already explain themselves, trivially obvious assignments.

Return ONLY a valid JSON object. Keys are line numbers as strings. Values are
explanation strings. Omit lines that need no explanation.
```

**Claude prompt (user):**
```
Language: {lang}

{numbered source lines}
```

**Implementation notes:**
- Use `@anthropic-ai/sdk` (already a project dependency or add as devDependency)
- Model: `claude-haiku-4-5-20251001` (fast, cheap, sufficient for code annotation)
- Temperature: 0.2 (consistent, factual explanations)
- Max tokens: 1024 for small files, 2048 for files > 50 lines
- Detect `lang` from file extension using the same `mapLangToVariant` table as the Snippet component (copy/import the mapping)
- If Claude returns invalid JSON, retry once; if still invalid, exit with error code 1

### `scripts/bulk-eli5.mjs`

Batch version. Scans all MDX files in `blog/`, finds `<Snippet filename="...">` usages, and generates `.eli5.json` for each referenced source file.

**Usage:**
```bash
node scripts/bulk-eli5.mjs [--force] [--dir blog/]
```

**Behavior:**
1. `find blog/ -name "*.mdx"` → list of MDX files
2. For each MDX: regex-scan for `<Snippet[^>]+filename="([^"]+)"` → collect filenames
3. Resolve each filename relative to the MDX file's directory
4. Deduplicate
5. For each unique source path: call `generate-eli5.mjs` logic (imported as a function, not spawned as child process)
6. Print summary: `generated: N, skipped: M, errors: K`

**Add to `package.json` scripts:**
```json
{
  "eli5": "node scripts/generate-eli5.mjs",
  "eli5:bulk": "node scripts/bulk-eli5.mjs"
}
```

## Files to modify

### `src/components/Snippet/index.js`

**New prop:** `eli5` — object mapping line-number strings to explanation strings. Passed directly from the imported JSON's `.explanations` field.

```jsx
// In MDX:
import eli5Data from './Dockerfile.eli5.json';
<Snippet filename="Dockerfile" code={rawDockerfile} eli5={eli5Data.explanations} />
```

**New state:**
```js
const [activeEli5Line, setActiveEli5Line] = useState(null);
```

**New rendered output when `eli5` is provided:**

Replace the `codeBlock` constant with a custom renderer that calls `renderEli5Block(code, lang, eli5)` instead of `<CodeBlock>`.

```jsx
function renderEli5Block(code, lang, eli5, activeEli5Line, setActiveEli5Line) {
  const lines = code.split('\n');
  // Use Prism to highlight the full block, then split by newline to get per-line HTML
  const grammar = Prism.languages[lang] || Prism.languages.plaintext;
  const highlighted = Prism.highlight(code, grammar, lang);
  const highlightedLines = highlighted.split('\n');

  return (
    <pre className={clsx('prism-code', `language-${lang}`, styles.eli5_pre)}>
      <code>
        {lines.map((_, i) => {
          const lineNum = String(i + 1);
          const hasExplanation = eli5 && eli5[lineNum];
          const isActive = activeEli5Line === lineNum;

          return (
            <div key={i} className={styles.eli5_line}>
              <span
                className={styles.eli5_code}
                dangerouslySetInnerHTML={{ __html: highlightedLines[i] || ' ' }}
              />
              {hasExplanation && (
                <span className={styles.eli5_badge_wrapper}>
                  <button
                    className={clsx(styles.eli5_badge, isActive && styles.eli5_badge_active)}
                    aria-label={`Explain line ${lineNum}`}
                    aria-expanded={isActive}
                    onClick={() => setActiveEli5Line(isActive ? null : lineNum)}
                    onMouseEnter={() => setActiveEli5Line(lineNum)}
                    onMouseLeave={() => setActiveEli5Line(null)}
                  >
                    ?
                  </button>
                  {isActive && (
                    <span role="tooltip" className={styles.eli5_tooltip}>
                      {eli5[lineNum]}
                    </span>
                  )}
                </span>
              )}
            </div>
          );
        })}
      </code>
    </pre>
  );
}
```

**Import Prism:**
```js
import Prism from 'prismjs';
```

Prism is already bundled by Docusaurus; this import is safe.

**Prop types addition:**
```js
eli5: PropTypes.objectOf(PropTypes.string),
```

**Note:** The ELI5 renderer only activates when both `code` (string) and `eli5` props are provided. If only `children` is provided (the MDX code fence pattern), the feature does not activate — this is an accepted limitation since line-by-line rendering requires the raw string.

## Known limitations (accepted)

- **Requires `code` prop** — the ELI5 renderer needs the source as a plain string. Snippets using only `children` (rendered MDX code fences) are not supported without refactoring the child extraction logic.
- **Prism language coverage** — languages not registered in the Docusaurus Prism bundle will fall back to plain text highlighting. Acceptable for the target file types (Dockerfile, YAML, Bash are all in the default bundle).
- **Tooltip position near bottom of viewport** — tooltip appears below the `?` badge. No viewport-aware repositioning in V1 (add as polish follow-up using `getBoundingClientRect`).
- **Line wrapping** — if a very long line wraps visually, the `?` badge still appears at the end of the first visual row. The tooltip is still correct. Acceptable.
- **Stale JSON** — if the source file changes after `.eli5.json` is generated, the explanations may be stale. Run `yarn eli5 <file> --force` to regenerate. No automatic staleness detection in V1.

## Verification checklist

1. `node scripts/generate-eli5.mjs blog/some-post/Dockerfile` → `Dockerfile.eli5.json` created
2. Run again (no `--force`) → script logs "skipped (already exists)" and exits 0
3. Run again with `--force` → file is regenerated with a new `generated` timestamp
4. Add import + `eli5` prop to the MDX file → `yarn start`
5. Open the Snippet block → `?` badges visible on annotated lines; no badges on blank/trivial lines
6. Hover `?` → tooltip appears with the explanation text
7. Move mouse away → tooltip disappears
8. Tab to `?` button → tooltip appears (keyboard accessible)
9. Click `?` on mobile (touch) → tooltip toggles on/off
10. `yarn build` → no build error; `.eli5.json` file is included in the bundle

## Deployment notes

No server-side component. The `.eli5.json` files are static assets bundled at build time.

**Workflow for new blog posts:**
```bash
# 1. Write the post and add source files
# 2. Generate ELI5 metadata
yarn eli5 blog/2026-xx-xx-my-post/Dockerfile
yarn eli5 blog/2026-xx-xx-my-post/compose.yaml

# 3. Import JSON in MDX and add eli5 prop to <Snippet>
# 4. Preview
yarn start

# 5. Commit both the source file AND the .eli5.json
git add blog/2026-xx-xx-my-post/Dockerfile.eli5.json
```

The `.eli5.json` files should be committed to the repo so CI/CD builds do not need an `ANTHROPIC_API_KEY`.
