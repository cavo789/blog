## KEY REVIEW DIMENSIONS

Focus your analysis on these high-level principles. Use the CORE MANDATE as your final validation check.

### 1. Strict Typing & Type Safety

- **Strict Types Declaration is Absolute:** Every PHP file containing code MUST start with `declare(strict_types=1);`. No exceptions.
- **Exhaustive Typing:** Every property, parameter, and return type MUST be explicitly typed. This includes using modern union types (`string|int`), intersection types (`Countable&Iterator`), and `never` where appropriate. Use of `mixed` is forbidden without explicit justification.
- **Strict Comparisons:** Loose comparisons (`==`, `!=`) are REJECTED. All comparisons MUST be strict (`===`, `!==`) to avoid type juggling bugs.
- **No Property Access Magic:** The use of magic methods like `__get`, `__set`, and `__call` is a strong code smell and is REJECTED unless used within a well-justified framework context (e.g., ORM entity).

### 2. Clean Architecture & SOLID Principles

- **Dependency Injection is Mandatory:** The `new` keyword is forbidden inside services, controllers, or any application logic class. All dependencies MUST be injected via the constructor. Direct instantiation couples the code and makes it untestable.
- **Single Responsibility:** Each class must have one, and only one, reason to change. Reject "Manager" or "Helper" classes that aggregate unrelated functionality.
- **Favor Composition Over Inheritance:** Classes should be `final` by default. Inheritance should be used sparingly and with clear justification.
- **Method Length:** Methods must not exceed 15 lines of logic. Complex operations must be refactored into smaller, private methods with descriptive names.

### 3. Modern PHP Best Practices (8.2+)

- **Constructor Property Promotion is Mandatory:** Use constructor property promotion for all DTOs and value objects. It is the modern standard for reducing boilerplate.
- **Immutability by Default:** Properties and classes MUST be declared `readonly` wherever possible. This prevents unexpected state changes and makes the code easier to reason about.
- **Enums Over Constants:** For sets of related values, `enum` (especially backed enums) MUST be used instead of class constants.
- **No Global State:** The use of the `global` keyword or the `$GLOBALS` superglobal is an architectural failure and is STRICTLY FORBIDDEN.
- **`die()` and `exit()` are Forbidden:** These functions halt execution and break the request/response lifecycle. They are only permissible in a front controller (`index.php`) or a CLI entry-point script for bootstrap failures.

### 4. Readability & Maintainability

- **PSR-12 Compliance:** All code must adhere to the PSR-12 coding style guide for naming conventions (PascalCase for classes, camelCase for methods).
- **Clarity Over Brevity:** The use of the null safe operator (`?->`) or complex `match` expressions is encouraged, but not at the expense of readability.
- **PHPDoc for Public APIs:** All public methods and properties must have a clear PHPDoc block explaining their purpose, parameters, and return values.

### 5. Security

- **No Raw SQL:** Reject any code that builds SQL queries by concatenating strings. All database interactions MUST use prepared statements via PDO or an abstraction layer (like Doctrine DBAL/ORM).
- **Output Escaping:** Any variable outputted in an HTML context MUST be escaped using `htmlspecialchars()` or a templating engine (like Twig) that provides auto-escaping. `echo $variable;` is a potential XSS vulnerability.
- **No `eval()`:** The use of `eval()` is a critical security risk and is STRICTLY FORBIDDEN.
- **No Hardcoded Secrets:** Reject any visible credentials, API keys, or passwords.
